Prop Drilling vs. Context API

Prop Drilling is the process of passing data from a parent component down through multiple layers of nested child components to reach a deeply nested one that needs it. This can become cumbersome.

The Context API provides a way to share values like these between components without having to explicitly pass a prop through every level of the tree.

Avoiding Unnecessary Renders

React components re-render when their state or props change. Sometimes, a component re-renders even if its output will be the same, which is inefficient. We can prevent this using techniques like React.memo for components, and useCallback/useMemo for functions and values.

Lazy Loading and Caching

Lazy Loading (or code-splitting) is the practice of splitting your app's code into smaller chunks which are loaded on demand, rather than loading everything upfront. This improves initial load time.

Caching involves storing data that has been fetched so it doesn't need to be fetched again. Libraries like React Query excel at this, managing cache invalidation, background updates, and more.

Yarn vs NPM vs PNPM

These are all package managers for JavaScript, used to install, manage, and share code packages (dependencies) for your projects. While they serve the same core purpose, they have key differences in performance, disk space usage, and dependency management.

Feature NPM (Node Package Manager) Yarn PNPM (Performant NPM)

Vite vs Create React App (CRA) vs Next.js

These are tools to build React applications, but they serve different purposes.

  • Create React App (CRA): Was the official scaffolding tool for React. It provided a zero-config setup for building client-side SPAs. It was sunsetted in 2023 because it became slow and inflexible compared to modern alternatives. Its use of Webpack for bundling and Jest for testing was powerful but often sluggish for development.
  • Vite: A modern build tool that focuses on speed. It uses the browser's native ES modules during development for instant server start and lightning-fast Hot Module Replacement (HMR). For production builds, it uses Rollup, which is highly optimized. Vite is a fantastic replacement for CRA for building SPAs.
  • Next.js: A full-stack React framework. It goes beyond a build tool, offering file-based routing, server-side rendering (SSR), static site generation (SSG), and API routes out of the box. It's ideal for complex applications, SEO-critical sites, and when you need both a frontend and backend in one project.

Issue: Vitest vs. Jest with Static Font Files

A common issue, like the one with @connect/typography using a .woff2 font file, highlights a key difference between test runners.

  • Why Jest handles it: Jest, commonly used with CRA, runs in a Node.js environment, not a real browser. It uses "transforms" to process non-JavaScript files. A default CRA setup includes a transform that replaces file imports (like fonts, CSS, images) with a mock string or object. So, Jest never actually tries to *load* the font file.
  • Why Vitest has issues: Vitest, built on Vite, has a different architecture. It can run tests in a Node environment or a browser-like environment (via `jsdom` or `happy-dom`). When it encounters a direct import of a static asset like a `.woff2` file within a component, it might try to resolve and process that file. If not configured correctly, this can fail, as the test environment doesn't know what to do with a binary font file.

Solution: Mocking in Vitest

The standard solution is to mock the problematic module. You tell Vitest: "whenever you see an import from `@connect/typography`, don't actually load it. Instead, use this fake object I'm providing." This isolates the test from the implementation detail of the font file.

// In your test setup file or at the top of a test file
import { vi } from 'vitest';

// Mock the entire package
vi.mock('@connect/typography', () => {
  // Return a mock object. Here, we mock the 'Typography' component
  // as a simple div. This prevents the test from breaking.
  return {
    Typography: (props) => <div>{props.children}</div>,
    // You can add other named exports as needed
  };
});

SPA vs MPA

Single-Page Applications (SPAs) load a single HTML page and dynamically update content as the user interacts with the app. Navigation is handled by JavaScript. (e.g., Gmail, Facebook).

Multi-Page Applications (MPAs) are the traditional web model where clicking a link requests a completely new page from the server. (e.g., Amazon, Wikipedia).

Aspect SPA (e.g., React, Vue) MPA (e.g., traditional server-rendered sites)
User Experience Faster, fluid, app-like feel after initial load. No page reloads. Slower transitions between pages due to full page reloads.
Initial Load Can be slow as the entire framework and app code needs to be loaded. Faster initial load for a single page.
SEO Can be challenging as search engine crawlers may not execute JavaScript correctly. Requires solutions like SSR. Excellent for SEO as each page is distinct and easily crawlable.
Development Complex state management. Clear separation between frontend and backend (API). Simpler state management (per page). Tightly coupled frontend and backend.

Major UI CSS Concepts

Flexbox

Flexbox is a one-dimensional layout model that offers an efficient way to lay out, align, and distribute space among items in a container, even when their size is unknown or dynamic.

  • Main Axis & Cross Axis: The primary concept. If `flex-direction` is `row` (default), the main axis is horizontal and the cross axis is vertical. If `column`, they are switched.
  • Container Properties:
    • display: flex;: Defines a flex container.
    • justify-content: Aligns items along the **main axis**.
    • align-items: Aligns items along the **cross axis**.
    • flex-wrap: Allows items to wrap onto multiple lines.
    • gap: Defines the space between flex items.

Spacing: Padding vs. Margin vs. Gap

These three properties define space in CSS, but in different ways.

  • Margin: Space *outside* an element's border. It pushes other elements away.
  • Padding: Space *inside* an element's border, between the border and the content.
  • Gap: A property for Flexbox and Grid containers to define the space *between* child elements, simplifying spacing without using margins.

Units: em vs px vs rem

Choosing the right CSS unit is crucial for creating scalable and maintainable designs.

  • px (Pixel): An absolute unit. 1px is one pixel on the screen. It's fixed and not scalable, which can be bad for accessibility. Good for things that should never change size, like border widths.
  • em: A relative unit. It's relative to the `font-size` of its direct parent element. If a parent's font-size is 16px, `1.5em` is 24px. This can lead to compounding effects in nested elements.
  • rem (Root em): A relative unit. It's relative to the `font-size` of the root element (the `` tag). This avoids the compounding issue of `em` and is excellent for creating scalable layouts. Changing the root font size scales the entire application proportionally. This is the preferred unit for font sizes and spacing in modern web design.
